⚠️ 分布式物理时钟倾斜导致消息 Seq 乱序 (Clock Skew Inversion)
风险描述：Nexus 的 SeqAllocator 是本地内存生成器。多台 Message 服务实例同时消费 JetStream 中同一会话的消息时，如果实例 A 的物理时钟比实例 B 慢 100ms：
实例 B 消费了消息 M1（物理时间 $T$），分配了 TS = $T$ 的 Seq。
实例 A 消费了消息 M2（物理时间 $T + 10ms$），但因为时钟落后，分配了 TS = $T - 90$ 的 Seq。
客户端收到的消息 Seq 将出现逆向回退，导致历史记录展示乱序，或拉取增量消息时发生丢包。
严重级别：High
2. ⚠️ 慢写场景下 SeqSyncer 内存队列满导致的 timeline 丢失 (Backpressure Drop Risk)
风险描述：当 MySQL/Redis 发生高延迟或网络抖动时，SeqSyncer 缓冲队列（32768）会迅速打满。
逻辑缺陷：在队列满时，主处理 Worker 会阻塞 200ms；超时后，系统将直接丢弃当前会话的 PushSeqUpdate 事件。
虽然消息已写入 MongoDB（保障消息不丢），但会话表的 actual_seq 滞后、Redis Timeline 未能更新。
业务后果：用户在客户端的会话列表中，该会话可能无法置顶、看不到最新消息预览，或者未读计数未能正确增加。如果此后该会话无新消息，此 stale 状态将长久保持。
严重级别：Medium-High
3. ⚠️ 路由表设计与 Redis Cluster 部署模式不兼容 (Redis Cluster Incompatibility)
风险描述：在 pkg/routing/user.go 中，lookupUserScript Lua 脚本会动态拼接 nodeKeyPrefix 与 node。
lua

local alive = redis.call('EXISTS', ARGV[1] .. node)
在 Redis Cluster 下，多键操作（如 userRouteKey(userID) 与 nodeKeyPrefix + node）如果不在同一个 Hash Slot（哈希槽），会触发 CROSSSLOT Keys in request don't hash to the same slot 错误，导致脚本直接执行失败。
限制了路由表的大规模水平扩容。
严重级别：Medium
